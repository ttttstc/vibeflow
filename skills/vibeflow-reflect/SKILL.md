# VibeFlow 回顾阶段

交付后闭环。创建回顾和下一迭代输入。

**开始时宣布：** "我正在使用 vibeflow-reflect skill 运行回顾。"

## 目的

发布后捕获学习：
- 什么工作得好
- 什么失败了
- 下一迭代应该改变什么
- 下一周期的具体改进

## 先决条件

运行此 skill 之前：
- 发布已完成（vibeflow-ship 完成）
- 所有功能完成
- 任务进度可用

## 步骤 1: 收集会话数据

### 1.1 读取任务进度

从 `task-progress.md`:
- 会话日志条目
- 已完成的功能
- 遇到的问题
- 时间估算 vs 实际

### 1.2 读取功能列表

从 `feature-list.json`:
- 所有具有最终状态的功能
- 花费比预期长的功能
- 范围变更的功能

### 1.3 审查之前的回顾

如果之前的回顾存在于 `.vibeflow/retro-YYYY-MM-DD.md`:
- 记录之前回顾中的行动项
- 检查它们是否已实施
- 结转未完成的项目

### 1.4 审查发布说明

从 `RELEASE_NOTES.md`:
- 交付了什么
- 已知限制
- 破坏性变更

## 步骤 2: 分析什么工作得好

### 2.1 识别成功

对于每个正面模式：
- 发生了什么
- 为什么它有效
- 如何复制

要考虑的类别：
- **流程**: 计划、审查、测试
- **架构**: 起作用的设计决策
- **团队**: 协作、沟通
- **工具**: 自动化、脚本、skill

### 2.2 记录成功

```
### 什么工作得好

#### 流程
- [成功 1]: 描述什么工作得好

#### 架构
- [成功 1]: 描述

#### 团队
- [成功 1]: 描述

#### 工具
- [成功 1]: 描述
```

## 步骤 3: 分析什么失败了

### 3.1 识别失败

对于每个问题或失败：
- 发生了什么
- 根本原因
- 影响

要考虑的类别：
- **流程**: 计划差距、审查失败
- **架构**: 导致问题的设计决策
- **团队**: 沟通问题、技能差距
- **工具**: 缺失的自动化、破碎的流程

### 3.2 记录失败

```
### 什么失败了

#### 流程
- [失败 1]: 描述什么出了问题
  - 根本原因: 为什么发生
  - 影响: 什么效果

#### 架构
- [失败 1]: 描述
  - 根本原因: ...
  - 影响: ...

#### 团队
- [失败 1]: 描述
  - 根本原因: ...
  - 影响: ...

#### 工具
- [失败 1]: 描述
  - 根本原因: ...
  - 影响: ...
```

## 步骤 4: 识别行动项

### 4.1 流程改进

对于每个识别的改进：
```
#### 行动: [简短标题]
**负责人**: 谁来负责
**什么**: 要做的具体变更
**何时**: 下一次迭代开始前
**成功标准**: 如何知道它完成了
```

### 4.2 架构改进

对于架构问题：
```
#### 行动: [简短标题]
**负责人**: 谁来负责
**什么**: 具体的重构或设计变更
**何时**: 在下一次迭代中
**成功标准**: 如何验证
```

### 4.3 工具改进

对于工具/流程差距：
```
#### 行动: [简短标题]
**负责人**: 谁来负责
**什么**: 要创建的新自动化或流程
**何时**: 下一次迭代开始前
**成功标准**: 如何验证
```

## 步骤 5: 创建回顾文档

### 5.1 创建回顾文件

文件: `.vibeflow/retro-YYYY-MM-DD.md`

```markdown
# 回顾 — YYYY-MM-DD

## 摘要
- **交付的功能**: N
- **总时间**: X 天
- **关键成功**: 1-2 句话总结
- **关键失败**: 1-2 句话总结

---

## 什么工作得好

### 流程
- [成功 1]
- [成功 2]

### 架构
- [成功 1]
- [成功 2]

### 团队
- [成功 1]
- [成功 2]

### 工具
- [成功 1]
- [成功 2]

---

## 什么失败了

### 流程
- [失败 1]
- [失败 2]

### 架构
- [失败 1]
- [失败 2]

### 团队
- [失败 1]
- [失败 2]

### 工具
- [失败 1]
- [失败 2]

---

## 行动项

| # | 行动 | 负责人 | 何时 | 完成 |
|---|--------|-------|------|------|
| 1 | 行动描述 | 姓名 | 下次之前 | [ ] |
| 2 | 行动描述 | 姓名 | 下个周期 | [ ] |

---

## 下一迭代输入

### 结转
- [之前回顾完成的项目]
- [之前回顾未完成的项目 — 原因]

### 新考虑
- [来自本周期的新见解]

### 范围建议
- [下一迭代范围的建议]
```

### 5.2 保存回顾文档

写入 `.vibeflow/retro-YYYY-MM-DD.md`。

## 步骤 6: 创建增量请求（如果需要）

### 6.1 识别后续工作

从失败和行动项：
- 需要的重构
- 要解决的技术债务
- 推迟的功能

### 6.2 创建增量请求

文件: `.vibeflow/increment-request.json`

```json
{
  "created": "YYYY-MM-DD",
  "source_retro": "retro-YYYY-MM-DD.md",
  "follow_up_work": [
    {
      "title": "工作描述",
      "category": "refactor|debt|deferred_feature",
      "priority": "high|medium|low",
      "reason": "为什么需要这个",
      "estimated_impact": "什么改进"
    }
  ],
  "process_improvements": [
    {
      "title": "描述",
      "action": "具体变更",
      "owner": "谁",
      "when": "何时"
    }
  ]
}
```

### 6.3 保存增量请求

写入 `.vibeflow/increment-request.json`。

## 步骤 7: 更新任务进度

在 `task-progress.md` 中：
- 标记回顾完成
- 链接到回顾文档
- 链接到增量请求（如果创建）

## 检查清单

在标记回顾完成之前：

- [ ] 所有会话数据已收集
- [ ] 成功已识别并记录
- [ ] 失败已识别并记录
- [ ] 根本原因已分析
- [ ] 行动项已创建并指定负责人
- [ ] 回顾文档已保存
- [ ] 增量请求已创建（如果需要）
- [ ] 任务进度已更新
- [ ] 下一迭代已通知

## 输出

| 文件 | 目的 |
|------|---------|
| `.vibeflow/retro-YYYY-MM-DD.md` | 回顾文档 |
| `.vibeflow/increment-request.json` | 下一迭代输入（如果需要） |

## 集成

**调用者:** `vibeflow` 路由器（发布完成后）
**需要:**
- `task-progress.md`（会话日志）
- `feature-list.json`（已完成的功能）
- 之前的回顾（如果存在）
- `RELEASE_NOTES.md`
**产出:**
- `.vibeflow/retro-YYYY-MM-DD.md` 中的回顾文档
- `.vibeflow/increment-request.json` 中的增量请求（如果需要）
**链接到:** 下一个 VibeFlow 周期（思考阶段）
