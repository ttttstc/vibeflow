# Frontend Design Constraints

这份约束只在 `vibeflow-design` 处理 **前端 / UI / 视觉界面** 时生效。

目标很简单：

- 有明确审美方向
- 不产出模板味界面
- 在设计阶段就说清楚字体、色彩、动效、布局
- 先暴露 tradeoff，再进入实现

## 开始前必须问清楚的 4 个问题

在写任何 UI/UX 方案前，先拿到这四类信息：

1. **谁在用，使用场景是什么**
2. **审美方向是什么**
3. **用户最后会记住什么**
4. **硬约束是什么**

要求：

- 审美方向必须具体命名，例如：dense editorial、raw terminal、warm analog、brutalist grid
- 禁止用 `clean and modern` 这类空泛描述
- 在进入 UI 设计前，用一句话复述最终选定的方向

## CSS / 样式策略

在开始任何前端设计前，必须明确单一策略：

- `Tailwind only`
- `CSS Modules only`
- `CSS-in-JS only`

不要混搭两套会冲突的样式系统。

## Typography

字体是最强的视觉决策之一。

要求：

- 明确 display font 和 text font 的分工
- display font 需要承载风格，不要隐形
- 不要把 `Inter`、`Roboto`、`system-ui` 当作主展示字体

## Color

颜色必须是系统，不是临时挑几组 hex。

要求：

- 用 token/变量描述色彩体系
- 通常采用一个主导色 + 一到两个强调色
- 所有正文对比度至少满足 WCAG AA

## Motion

动效要有预算，不要到处乱加。

要求：

- 重点放在首屏进入和关键状态变化
- entrance 200-400ms
- exit 150-250ms
- 明确是“必要反馈”还是“纯装饰”

## Layout

布局要有空间策略，不要自动回到模板。

要求：

- 说明是高密度还是留白型
- 背景应有气氛，不要默认纯白或纯黑平面
- 如果桌面设计语言和移动端约束冲突，要明确说明谁优先

## 常见冲突

这些组合要避免：

| 不要混用 | 原因 |
|---|---|
| Tailwind + CSS Modules 写在同一元素上 | 级联和维护都容易失控 |
| Framer Motion + CSS transition 同时驱动同一属性 | 容易双重动画和抖动 |
| 多套图标库混用 | 视觉风格不一致 |
| 多个 display font 并列竞争 | 风格相互抵消 |

## 常见模板味陷阱

提交设计前，自查是否不小心掉回这些默认模式：

- 紫蓝渐变铺白底
- 大标题 + 一行副标题 + 两个并排 CTA
- 一排阴影/圆角/间距完全一致的卡片
- 顶部导航：左 logo、中链接、右主按钮
- 区块轮流白底和浅灰底
- 图标在上、标题在中、描述在下的标准堆叠

如果这些元素是刻意选择，可以保留；如果只是默认滑过去了，就要重做。

## 最后测试

如果把当前内容整批换成另一套内容，这个页面布局仍然“完全合理”，那大概率做出来的是模板，不是设计。
