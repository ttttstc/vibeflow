# 实现计划编写指南

本文档描述如何在 VibeFlow 开发流程中编写高质量的实现计划，以及如何将 feature 拆解为可测试的步骤。

## 1. 计划与 feature-list.json 的关系

### 1.1 feature-list.json 结构
```json
{
  "features": [
    {
      "id": "feat-001",
      "name": "功能名称",
      "phase": "build-work",
      "status": "in-progress",
      "implementations": [
        {
          "step": 1,
          "description": "实现步骤描述",
          "verification": "验证方法"
        }
      ]
    }
  ]
}
```

### 1.2 计划与 feature 的映射
- 每个 feature 应该在 `implementations` 中有对应的实现步骤
- 实现计划是 `implementations` 数组的详细展开
- 计划完成后，feature 状态应更新为 `completed`

## 2. 如何将 feature 拆解为可测试的步骤

### 2.1 拆解原则
```
从外部接口到内部实现：
1. API 接口定义（输入、输出、错误处理）
2. 业务逻辑层
3. 数据访问层
4. 基础设施层（缓存、日志等）
```

### 2.2 拆解流程
1. **识别核心功能**: 这个 feature 最基本的输入输出是什么？
2. **识别边界条件**: 什么情况下会失败？边界值有哪些？
3. **识别依赖关系**: 需要调用哪些外部服务？依赖哪些数据？
4. **确定实施顺序**: 哪些可以并行？哪些必须串行？

### 2.3 步骤粒度
每个步骤应该满足：
- 可以独立实现和测试
- 有明确的完成标准
- 不会因为其他步骤的延迟而阻塞
- 耗时不超过半天（建议）

## 3. 验证点与测试用例的对应

### 3.1 验证点类型
| 验证类型 | 说明 | 对应测试 |
|----------|------|----------|
| 功能验证 | 核心功能是否正常工作 | 单元测试、集成测试 |
| 边界验证 | 边界条件和异常处理 | 边界测试、负向测试 |
| 性能验证 | 响应时间、资源消耗 | 性能测试 |
| 安全验证 | 权限、认证、数据保护 | 安全测试 |

### 3.2 验证点转测试
每个验证点应该有对应的测试用例：
```python
# 验证点：API 应该正确处理空输入
def test_api_handles_empty_input():
    """验证空输入返回适当错误"""
    response = api.call_with_empty_input()
    assert response.status_code == 400
    assert "error" in response.body
```

### 3.3 验证优先级
1. Happy Path（正常流程）— 必须首先验证
2. 常见边界条件 — 验证主要边界
3. 异常情况 — 验证错误处理
4. 性能要求 — 在功能稳定后验证

## 4. 实现计划模板

### 4.1 标准结构
```markdown
# 实现计划: [Feature 名称]

## Feature 信息
- ID: feat-xxx
- 负责人: [姓名]
- 开始日期: [日期]
- 预计完成: [日期]

## 目标概述
[简要描述这个 feature 要实现什么]

## 详细步骤

### Step 1: [步骤名称]
**目标**: [这个步骤要达成什么]

**实现内容**:
- [具体实现项 1]
- [具体实现项 2]

**验证方法**:
- [验证点 1，对应测试]
- [验证点 2，对应测试]

**完成标准**:
- [ ] 验证点 1 通过
- [ ] 验证点 2 通过
- [ ] 代码审查通过

### Step 2: ...
```

### 4.2 填写指南
- **目标**: 用一句话描述，清晰明确
- **实现内容**: 具体的技术实现，不模糊
- **验证方法**: 必须可执行、可观测
- **完成标准**: 有明确的检查项

## 5. 计划与 TDD 结合

### 5.1 TDD 循环
```
红（失败测试）-> 绿（最小代码通过）-> 重构
```

### 5.2 计划中嵌入 TDD
```markdown
### Step 1: 实现基础数据模型
**TDD 循环**:
1. 先写测试: test_data_model_creation()
2. 实现最小代码使测试通过
3. 如需要则重构

**测试用例**:
- test_data_model_creation
- test_data_model_validation
```

### 5.3 测试先行检查清单
- [ ] 每个步骤都有对应的测试
- [ ] 测试在实现前编写
- [ ] 测试覆盖正常和异常情况
- [ ] 测试可以自动化运行

## 6. 常见问题

### Q: 步骤太粗略怎么办？
A: 将步骤进一步拆分为子步骤，每个子步骤应该：
- 有明确的输入输出
- 可以单独测试
- 耗时不超过 2 小时

### Q: 验证点和测试的对应关系不清晰？
A: 使用表格明确对应：
| 验证点 | 测试函数 | 测试类型 |
|--------|----------|----------|
| API 返回正确格式 | test_api_response_format | 单元测试 |
| 空输入正确处理 | test_api_empty_input | 集成测试 |

### Q: 如何处理依赖外部服务的情况？
A:
1. 使用 Mock 隔离外部依赖
2. 计划中明确标注依赖项
3. 预留接口集成时间

## 7. 计划审查要点

在实现前审查计划：
- [ ] 每个步骤都有明确的验证方法
- [ ] 验证方法可以转化为自动化测试
- [ ] 步骤之间没有循环依赖
- [ ] 时间估计合理
- [ ] 风险点已识别并有应对方案

## 8. 进度跟踪

实现过程中：
- 每日更新步骤完成状态
- 记录遇到的问题和解决方案
- 及时调整后续计划
- 在 feature-list.json 中更新进度
