# 系统化调试流程指南

本文档描述在 VibeFlow 开发流程中进行系统化调试的方法，帮助开发者快速定位和解决问题。

## 1. 问题复现（最小化复现步骤）

### 1.1 收集环境信息
在开始调试前，收集以下信息：
- 操作系统和版本
- Python/Node/Java 环境版本
- 相关依赖版本
- 错误发生的具体操作步骤

### 1.2 创建最小复现用例
1. 剥离所有无关代码，只保留能触发问题的最小代码片段
2. 记录复现步骤，确保可以精确重复
3. 如果问题与特定数据相关，提取出导致问题的最小数据集

### 1.3 复现验证清单
```
[ ] 确认问题在干净环境中可复现
[ ] 记录触发条件（输入、参数、环境状态）
[ ] 确定问题发生的确定性（100% 或概率性）
```

## 2. 假设-验证循环

### 2.1 提出假设
基于错误信息和观察到的行为，提出可能的根本原因假设：
- 不要假设多个问题同时发生
- 每个假设应能解释观察到的所有症状

### 2.2 设计验证实验
针对每个假设，设计一个能证明或否定它的实验：
```
假设：如果 X 是问题，那么修改 Y 应该导致 Z 结果
验证：执行 Y 变更，观察是否出现 Z
```

### 2.3 优先级排序
按以下顺序验证假设：
1. 最可能的原因（基于错误信息、常见错误模式）
2. 最容易验证的原因（低成本实验）
3. 最危险的原因（可能导致数据丢失、安全问题）

## 3. 二分查找定位

### 3.1 代码层面
- 使用 git bisect 进行版本二分查找
- 注释掉一半代码，判断问题是否消失
- 递归缩小范围直至定位具体代码行

### 3.2 日志层面
- 使用二分查找定位问题代码段
- 添加日志标记，记录中间状态
- 对比正常和异常执行的日志差异

### 3.3 数据层面
- 使用二分搜索定位导致问题的数据项
- 对比有效数据和无效数据的差异

## 4. 调试日志使用

### 4.1 日志级别规范
```
DEBUG: 详细的执行流程信息（开发时使用）
INFO: 重要的状态变化和业务事件
WARNING: 潜在问题但不影响功能
ERROR: 错误信息，需要关注
CRITICAL: 严重问题，可能导致系统崩溃
```

### 4.2 有效日志实践
- 在关键路径上添加日志记录
- 记录足够的上下文信息（请求ID、用户ID等）
- 避免日志噪声（重复日志、无用信息）
- 关键操作使用结构化日志（JSON格式）

### 4.3 日志分析技巧
```bash
# 查找特定错误
grep -i "error" application.log

# 查找特定时间范围内的日志
grep "2024-01-15 10:00" application.log

# 分析错误模式
awk '/ERROR/ {print $NF}' application.log | sort | uniq -c
```

## 5. 与 TDD 结合的调试方法

### 5.1 红条先行
遇到 bug 时，先编写一个能够复现该 bug 的测试用例：
```python
def test_bug_reproduction():
    """复现 bug 的测试用例"""
    # 设置已知会导致 bug 的条件
    setup_buggy_condition()
    # 执行会触发 bug 的操作
    result = operation_that_triggers_bug()
    # 验证 bug 确实存在（测试应该失败）
    assert result == expected_correct_behavior  # 这会失败
```

### 5.2 隔离问题
- 将问题隔离到最小的代码单元
- 如果问题涉及多个模块，使用 mock 排除其他模块的影响

### 5.3 修复验证
修复后：
1. 原来的 bug 复现测试应该通过
2. 确保没有破坏现有功能（其他测试仍然通过）
3. 添加回归测试防止未来再犯

### 5.4 调试工作流
```
1. 编写失败测试（复现 bug）
2. 分析错误，找到根本原因
3. 实现修复
4. 运行测试，确认通过
5. 运行完整测试套件，确保无回归
6. 清理调试代码和临时日志
```

## 6. 常用调试工具

### Python
- `pdb` / `ipdb`: 交互式调试器
- `breakpoint()`: Python 3.7+ 内置断点
- `logging`: 日志模块
- `pytest.set_trace()`: pytest 调试钩子

### JavaScript/TypeScript
- Chrome DevTools: 浏览器调试
- Node.js 调试: `node inspect script.js`
- VS Code 调试器: 集成调试环境

### Java
- IDE 内置调试器（IntelliJ IDEA、Eclipse）
- JDB: 命令行调试器
- 日志框架: Log4j、SLF4J

## 7. 常见问题模式

| 模式 | 特征 | 排查方法 |
|------|------|----------|
| 空指针异常 | 变量未初始化 | 检查变量赋值路径 |
| 竞态条件 | 间歇性失败 | 增加日志，检查并发访问 |
| 内存泄漏 | 性能逐渐下降 | 使用 profiler 分析内存使用 |
| 配置错误 | 行为与预期不符 | 检查配置文件和环境变量 |
| 版本不兼容 | 特定环境下失败 | 对比环境差异 |

## 8. 调试检查清单

在宣布问题已修复前，确认：
- [ ] 问题在最小复现环境中不再出现
- [ ] 所有相关测试通过
- [ ] 没有引入新的问题
- [ ] 代码已清理（无调试代码、临时注释）
- [ ] 文档已更新（如有必要）
