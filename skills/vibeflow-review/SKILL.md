# VibeFlow 全局变更审查

在发布前对所有已完成的功能进行全面审查。跨功能分析结构问题、隐藏回归和完整性缺口。

**开始时宣布：** "我正在使用 vibeflow-review skill 进行全局变更审查。"

## 目的

在所有功能通过各自审查后，验证完整的变更集是否可以发布：
- 代码库的结构问题
- 功能交互带来的隐藏回归
- 集成层面缺失的测试
- 交付物中的完整性缺口

## 何时运行

- 所有功能通过各自的规格审查后
- 系统测试完成后（如果已启用）
- 发布前
- 由 `vibeflow` 路由器调用（构建和发布阶段之间）

## 步骤 1: 加载审查上下文

### 1.1 读取功能列表

从 `feature-list.json`:
- 所有 `"status": "passing"` 的功能
- 所有功能 ID、标题、类别
- 功能依赖关系

### 1.2 读取任务进度

从 `task-progress.md`:
- 已完成工作的会话日志
- 构建期间记录的任何问题
- 开放性问题或已知限制

### 1.3 读取所有规格文档

从 `docs/plans/`:
- 所有 SRS 部分
- 所有设计部分
- 所有计划文档

### 1.4 获取 Git 差异

```bash
git diff main...HEAD --stat
git diff main...HEAD
```

这将获取自基础分支以来的完整变更集。

## 步骤 2: 结构审查

### 2.1 架构完整性

审查完整变更集：
- [ ] 新模块/类遵循架构
- [ ] 跨功能依赖关系已管理
- [ ] 没有引入循环依赖
- [ ] 模块间的 API 契约已维护

### 2.2 代码组织

- [ ] 文件按约定组织
- [ ] 没有需要拆分的大型文件
- [ ] 跨功能没有重复代码
- [ ] 配置已正确外部化

### 2.3 依赖管理

- [ ] 新依赖关系已论证
- [ ] 依赖版本已锁定
- [ ] 没有有漏洞的依赖
- [ ] 依赖许可证可接受

## 步骤 3: 回归分析

### 3.1 功能交互

对于每一对可能交互的功能：
- [ ] 功能在一起正常工作
- [ ] 没有资源冲突
- [ ] 功能间没有数据损坏
- [ ] API 端点不冲突

### 3.2 集成点

- [ ] 所有集成点已测试
- [ ] 边界处的错误处理正常工作
- [ ] 功能间数据流正确
- [ ] 集成点没有数据丢失

### 3.3 副作用

- [ ] 变更没有意外的副作用
- [ ] 全局状态变更是有意的
- [ ] 配置变更向后兼容
- [ ] 对现有 API 没有破坏性变更

## 步骤 4: 测试覆盖率审查

### 4.1 单元测试覆盖率

从所有测试结果：
- [ ] 核心逻辑有足够的单元测试
- [ ] 边缘情况已覆盖
- [ ] 错误路径已覆盖

### 4.2 集成测试覆盖率

- [ ] 跨功能交互已测试
- [ ] 端到端场景已覆盖
- [ ] 集成点已执行

### 4.3 系统测试覆盖率

如果运行了系统测试：
- [ ] 所有系统测试用例通过
- [ ] NFR 已验证
- [ ] 性能可接受

## 步骤 5: 文档审查

### 5.1 代码文档

- [ ] 公共 API 已文档化
- [ ] 复杂逻辑已解释
- [ ] 新公共接口没有缺失的文档

### 5.2 用户文档

- [ ] 用户面向的变更已文档化
- [ ] API 变更已文档化（如果适用）
- [ ] 破坏性变更已文档化

### 5.3 发布说明

- [ ] 所有新功能已列出
- [ ] 错误修复已文档化
- [ ] 破坏性变更已突出显示
- [ ] 提供了迁移步骤（如果需要）

## 步骤 6: 安全审查

### 6.1 认证与授权

- [ ] 新端点已保护
- [ ] 权限正确执行
- [ ] 没有权限提升

### 6.2 输入验证

- [ ] 所有用户输入已验证
- [ ] 防止注入攻击
- [ ] 日志中已清理敏感数据

### 6.3 数据处理

- [ ] 敏感数据已加密
- [ ] PII 处理正确
- [ ] 遵循数据保留策略

## 步骤 7: 编译审查结果

### 7.1 问题列表

记录每个发现：
```
## 发现: [标题]
**严重性:** Critical | Important | Minor
**类别:** Structure | Regression | Coverage | Docs | Security
**描述:** 发现了什么
**证据:** 在哪里/如何检测到
**影响:** 可能出什么错
**修复:** 建议的修复
```

### 7.2 严重性分类

| 严重性 | 操作 | 阻塞发布? |
|--------|------|-----------|
| Critical | 立即修复 | Yes |
| Important | 发布前修复 | Yes |
| Minor | 下一版本修复 | No |

### 7.3 审查结论

**APPROVED** — 可以发布
**CONDITIONAL** — 带已知问题发布（已文档化）
**BLOCKED** — 发布前必须修复

## 步骤 8: 处理发现

### 8.1 修复关键问题

对于每个关键发现：
1. 在本次会话中修复
2. 重新运行相关测试
3. 验证修复
4. 记录解决方案

### 8.2 记录重要问题

对于每个重要发现：
1. 记录为已知限制
2. 如果面向客户则添加到发布说明
3. 添加到下一迭代的待办事项

### 8.3 更新任务进度

在 `task-progress.md` 中记录发现和解决方案。

## 检查清单

在标记全局变更审查完成之前：

- [ ] 所有功能已审查结构问题
- [ ] 功能交互已验证
- [ ] 集成测试足够
- [ ] 文档完整
- [ ] 安全审查通过
- [ ] 没有关键问题遗留
- [ ] 审查结论: APPROVED 或 CONDITIONAL

## 集成

**调用者:** `vibeflow` 路由器（构建完成后，发布前）
**需要:**
- `feature-list.json`（所有功能通过）
- `task-progress.md`（会话日志）
- `docs/plans/` 中的所有规格文档
- 来自基础分支的 Git 差异
- 系统测试结果（如果适用）
**产出:** 带发现的审查结论
**链接到:** `vibeflow-ship`（如果批准）
