## 目的

当工作流包含 UI 工作时，在 `docs/plans/*-ucd.md` 生成一份全面的界面和样式指导文档。本文档作为实施用户界面的权威参考，确保应用程序的一致性、可访问性和响应性。

UCD 捕获工作流规格中派生的所有 UI 需求，并将其转化为设计和实施团队的具体指导。

## 输入

UCD 生成器接受以下输入：

- **`.vibeflow/workflow.yaml`** - 定义工作流步骤、屏幕转换和 UI 需求的主要输入
- **`docs/plans/*-srs.md`** - 包含功能需求的软件需求规格说明书
- **`docs/plans/think-output.md`** - 包含用户旅程和交互模型的设计思维输出
- **`docs/plans/pood-output.md`**（可选）- 如果有的话，面向对象设计的输出
- **`docs/plans/adr-*.md`**（可选）- 影响 UI 选择的架构决策记录

## 前置条件

仅在以下情况下执行此 skill：
- `workflow.yaml` 中的工作流明确定义了 UI 交互（屏幕、表单、模态框、导航）
- 功能范围包括面向用户的界面元素
- 仅当模板和功能范围明显没有 UI 组件时才跳过

## 生成 `docs/plans/*-ucd.md` 的步骤

### 步骤 1：分析 workflow.yaml 中的 UI 需求

解析工作流文件并提取所有 UI 相关规格：

```
1.1. 识别带 UI 元素的屏幕/步骤定义
1.2. 记录屏幕转换和导航流程
1.3. 提取用户输入需求（表单、字段、验证）
1.4. 识别数据显示需求（列表、详情、图表）
1.5. 记录模态框/对话框交互
1.6. 记录通知和反馈需求
```

### 步骤 2：清点 UI 元素

创建所有界面组件的全面清单：

```
2.1. 列出所有表单，包含字段、类型和验证规则
2.2. 枚举所有按钮和交互控件
2.3. 记录导航组件（菜单、标签、面包屑）
2.4. 详细说明模态框和对话框需求
2.5. 编目数据显示组件（表格、卡片、图表）
2.6. 记录通知和状态指示器
```

### 步骤 3：定义响应式断点

根据工作流需求建立响应式设计策略：

```
3.1. 移动端优先断点：320px - 767px
3.2. 平板断点：768px - 1023px
3.3. 桌面端断点：1024px - 1439px
3.4. 宽桌面端：1440px 及以上
3.5. 记录每个断点适配的组件
3.6. 指定布局网格系统（推荐 12 列）
```

### 步骤 4：指定可访问性需求

为所有 UI 元素定义可访问性标准：

```
4.1. WCAG 2.1 AA 级合规目标
4.2. 颜色对比度（正常文本 4.5:1，大文本 3:1）
4.3. 焦点指示器需求
4.4. 交互元素的 ARIA 标签需求
4.5. 所有工作流的键盘导航路径
4.6. 屏幕阅读器兼容性说明
```

### 步骤 5：生成输出文档

使用以下模板将所有收集的信息组装成标准化的 UCD 格式。

## UI 元素清单结构

### 表单

| 元素 | 类型 | 验证 | 必填 | 备注 |
|------|------|------|------|------|
| 字段名 | text/email/password/number/date | 正则表达式或范围 | yes/no | 描述 |

### 按钮

| 按钮 | 操作 | 状态变体 | 键盘快捷键 |
|------|------|----------|------------|
| 标签 | submit/cancel/delete/navigate | default/hover/active/disabled/loading | Ctrl+S 等 |

### 导航

| 组件 | 类型 | 项目 | 行为 |
|------|------|------|------|
| 主菜单 | sidebar/topbar/dropdown | 项目列表 | 移动端折叠 |
| 面包屑 | static/dynamic | 路径项目 | 点击导航 |
| 标签 | horizontal/vertical | 标签名称 | 键盘箭头导航 |

### 模态框和对话框

| 模态框 | 触发器 | 尺寸 | 关闭行为 |
|--------|--------|------|----------|
| 名称 | 操作 | sm/md/lg/fullscreen | X按钮/Escape/外部点击 |

### 数据显示

| 组件 | 数据类型 | 分页 | 可排序 |
|------|----------|------|--------|
| 表格 | 数组 | yes/no | 列名 |
| 卡片网格 | 项目 | infinite/numbered | no |
| 详情视图 | 单个项目 | n/a | n/a |

## 可访问性检查清单

### WCAG 2.1 AA 级基础

- [ ] 所有图片有 alt 文本（信息性图片非空，装饰性图片为空）
- [ ] 颜色不是传递信息的唯一手段
- [ ] 正常文本对比度至少 4.5:1，大文本 3:1
- [ ] 文本可放大至 200% 而不丢失功能
- [ ] 无内容每秒闪烁超过 3 次
- [ ] 所有功能可通过键盘使用
- [ ] 无键盘陷阱
- [ ] 提供跳过导航链接
- [ ] 语言属性正确设置
- [ ] 焦点顺序逻辑且直观

### 键盘导航

- [ ] Tab 按逻辑顺序在所有交互元素间移动
- [ ] Enter/Space 激活按钮和链接
- [ ] Escape 关闭模态框和下拉菜单
- [ ] 箭头键在组件内导航（菜单、标签、网格）
- [ ] 焦点有视觉指示（轮廓或高亮）
- [ ] 焦点不会意外移动或禁用

### 屏幕阅读器支持

- [ ] 定义 ARIA 地标（banner、navigation、main、contentinfo）
- [ ] 表单输入有关联标签
- [ ] 错误消息被宣布并关联到字段
- [ ] 动态内容更改通过实时区域宣布
- [ ] 模态框焦点被捕获和管理
- [ ] 标题遵循正确层级（h1 -> h2 -> h3）

## 集成点

### 链自

1. **需求分析**（`*-srs.md`）：UCD 继承功能需求并将其转化为 UI 规格
2. **设计思维**（`think-output.md`）：UCD 整合用户旅程地图和交互模型
3. **工作流定义**（`workflow.yaml`）：UCD 提取屏幕定义、导航流程和交互需求

### 流入

1. **设计阶段**：为 mockup 和原型创建提供具体规格
2. **实施阶段**：作为组件开发和集成的参考
3. **测试阶段**：提供 UI 测试和可访问性验证的验收标准
4. **文档**：为用户帮助文档和入职流程形成基础

### 工作流触发条件

在以下情况下调用 UCD 生成：
- `workflow.yaml` 包含带 `ui: true` 或 `screen:` 定义的步骤
- 任务卡片引用界面更改
- 功能开发包含新的或修改的面向用户的组件

## 质量标准

生成的 UCD 在以下情况下视为完整：
- 所有工作流屏幕都有相应的组件定义
- 所有表单包含字段级验证规格
- 所有交互元素都有键盘和可访问性需求
- 每个组件类型都定义了响应式行为
- 样式说明为实施一致性提供了足够指导
