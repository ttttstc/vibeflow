# VibeFlow 系统测试

在构建完成后运行系统级测试。验证所有功能的集成行为。

**开始时宣布：** "我正在使用 vibeflow-test-system skill 运行系统级测试。"

## 目的

验证完整系统正常工作：
- 所有功能集成在一起
- 端到端场景正常工作
- 满足非功能性需求
- 没有引入回归

## 何时运行

- 所有单个功能通过审查后
- 构建工作完成后
- 发布前
- 由 `vibeflow-build-work` 调用（所有功能通过时）或 `vibeflow` 路由器调用

## 先决条件

运行此 skill 之前：
- `feature-list.json` 中所有功能标记为 `"status": "passing"`
- 系统文档可用
- 测试环境就绪

## 步骤 1: 加载系统上下文

### 1.1 读取功能列表

从 `feature-list.json`:
- 所有已完成的功能
- 功能类别
- 功能集成点

### 1.2 读取系统文档

从 `docs/plans/`:
- 系统架构
- 集成点
- 数据流
- 外部依赖

### 1.3 读取需求

从 `docs/plans/*-srs.md`:
- NFR-xxx（非功能性需求）
- 系统级验收标准
- 性能需求
- 安全需求

### 1.4 读取设计文档

从 `docs/plans/*-design.md`:
- 系统架构
- 组件交互
- 外部服务集成

## 步骤 2: 识别系统测试场景

### 2.1 端到端场景

对于每个主要用户旅程：
- [ ] 完整的用户工作流已测试
- [ ] 路径中的所有功能已验证
- [ ] 数据端到端正确流动
- [ ] 错误处理正常工作

### 2.2 集成点

对于每个集成：
- [ ] 数据交换已验证
- [ ] 边界处的错误处理
- [ ] 超时行为正确
- [ ] 重试逻辑正常工作

### 2.3 NFR 验证

对于每个 NFR：
- [ ] 性能标准可测量
- [ ] 安全标准可测试
- [ ] 可扩展性标准可验证
- [ ] 可靠性标准可证明

## 步骤 3: 创建系统测试用例

### 3.1 定义测试用例

**用例 ID:** `ST-SYS-{SEQ}`

**类别:**

| 类别 | 描述 |
|----------|-------------|
| `e2e` | 端到端用户场景 |
| `integration` | 组件集成 |
| `performance` | NFR 性能测试 |
| `security` | NFR 安全测试 |
| `regression` | 之前修复的 bug |

### 3.2 最低系统测试

每个系统测试周期必须包括：

**端到端测试（至少）：**
- 主要用户旅程测试
- 次要用户旅程测试
- 错误恢复旅程测试

**集成测试（至少）：**
- 所有外部服务集成已测试
- 所有内部组件集成已测试

**NFR 测试（如果指定）：**
- 性能测试（响应时间 < 阈值）
- 负载测试（并发用户）
- 安全测试（auth/authz 验证）

## 步骤 4: 执行系统测试

### 4.1 启动测试环境

1. 启动所有需要的服务
2. 验证健康端点
3. 确保测试数据可用
4. 记录 PID 用于清理

### 4.2 运行端到端测试

对于每个 E2E 场景：
1. 执行完整的用户旅程
2. 验证所有步骤成功完成
3. 验证最终数据完整性
4. 记录 PASS/FAIL

### 4.3 运行集成测试

对于每个集成点：
1. 触发集成
2. 验证数据交换
3. 验证错误处理
4. 记录 PASS/FAIL

### 4.4 运行 NFR 测试

对于每个 NFR 需求：
1. 执行性能/负载/安全测试
2. 对照阈值测量
3. 记录指标值
4. 记录 PASS/FAIL

### 4.5 运行回归测试

对于每个之前修复的 bug：
1. 执行回归场景
2. 验证修复仍然有效
3. 没有新的副作用
4. 记录 PASS/FAIL

## 步骤 5: 处理失败

### 5.1 任何测试失败

记录失败：
```
## 失败: [测试用例 ID]
**类型:** E2E | Integration | NFR | Regression
**描述:** 什么失败了
**预期:** 应该发生什么
**实际:** 发生了什么
**影响:** 系统影响
**修复:** 需要的修复
```

### 5.2 修复或升级

**对于关键失败：**
1. 修复实现
2. 重新运行失败的测试
3. 重新运行相关测试
4. 不要跳过或绕过

**对于 NFR 失败：**
1. 记录差距
2. 确定是否可以发布
3. 如果阻塞则升级给用户

## 步骤 6: 编译结果

### 6.1 测试摘要

```
## 系统测试摘要

**日期**: YYYY-MM-DD
**总测试数**: N
**通过**: N
**失败**: N

### 按类别的结果
| 类别 | 总数 | 通过 | 失败 |
|----------|-------|------|------|
| E2E | N | N | N |
| Integration | N | N | N |
| Performance | N | N | N |
| Security | N | N | N |
| Regression | N | N | N |
```

### 6.2 创建报告

文件: `docs/plans/*-st-report.md`

```markdown
# 系统测试报告 — YYYY-MM-DD

## 摘要
- **系统**: [项目名称]
- **版本**: [版本]
- **日期**: YYYY-MM-DD
- **总体状态**: PASS | FAIL

## 测试结果

### 端到端测试
[结果表]

### 集成测试
[结果表]

### NFR 测试
[带指标的结果表]

### 回归测试
[结果表]

## 发现的问题

### 关键
[必须修复的问题]

### 重要
[应该修复的问题]

### 次要
[稍后处理的问题]

## 签收

- [ ] 系统测试完成
- [ ] 所有关键问题已解决
- [ ] 可以发布
```

### 6.3 保存报告

写入 `docs/plans/*-st-report.md`。

## 步骤 7: 完成测试

### 7.1 最终状态

如果所有测试通过：
- 系统测试完成
- 进入审查（vibeflow-review）

如果测试失败但有已知问题：
- 记录为有条件批准
- 进入审查

如果测试关键失败：
- 修复问题
- 重新运行系统测试

### 7.2 停止测试环境

1. 停止所有测试服务
2. 验证清理
3. 记录状态

## 检查清单

在标记系统测试完成之前：

- [ ] 所有 E2E 场景已测试并通过
- [ ] 所有集成点已测试并通过
- [ ] 所有 NFR 测试已执行并记录结果
- [ ] 所有回归测试通过
- [ ] 报告已保存到 `docs/plans/*-st-report.md`
- [ ] 测试环境已停止并清理
- [ ] 系统测试结论: PASS 或 CONDITIONAL

## 质量门

| 门 | 要求 |
|------|-------------|
| E2E 覆盖率 | 所有主要用户旅程已测试 |
| 集成覆盖率 | 所有集成点已测试 |
| NFR 验证 | 所有 NFR 需求已验证 |
| 回归 | 所有之前修复的 bug 已验证 |

## 集成

**调用者:** `vibeflow` 路由器（所有功能通过后）或 `vibeflow-build-work`（步骤 7）
**需要:**
- 所有功能 `"status": "passing"`
- `docs/plans/` 中的系统文档
- 带 NFR 的 SRS
- 测试环境就绪
**产出:**
- `docs/plans/*-st-report.md` 中的系统测试报告
- 测试执行证据
**链接到:** `vibeflow-review`（如果测试通过）
