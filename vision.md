# VibeFlow Vision

## 定位

VibeFlow 的目标是让 agent 用对方法，而不是替 agent 再写一个执行内核。

VibeFlow 应该做的是 control plane，不是 another runtime。

Claude、Codex、CodeBuddy 这类 agent runtime 已经擅长：

- 阅读上下文并推理
- 拆解任务并调整计划
- 编辑代码并运行命令
- 调用子代理协作
- 根据结果动态修正执行路径

VibeFlow 不应该重复建设这些通用能力。

## 核心原则

### 1. 只把 100% 可机械化的东西写成脚本

适合脚本化的事情：

- 初始化项目结构
- 生成模板和配置
- 校验产物是否存在
- 运行确定性的规则检查
- 生成索引、摘要和报告
- 迁移、归档、版本和发布辅助

不适合脚本化的事情：

- 通用规划
- 通用推理
- 通用调试决策
- 通用子代理调度
- 依赖大量上下文判断的实施策略

### 2. 只把必须稳定复现的东西固化成状态机

状态机应服务于：

- 阶段定义
- 阶段切换条件
- 恢复点
- 审批点
- 必要产物和完成证据

状态机不应用来替代 agent 的正常思考过程，也不应用来编码过多临时策略。

### 3. 只把经常忘、忘了就出事的东西做成 gate

Gate 用来防止高代价失误，例如：

- 缺少关键设计或验证产物
- 跳过必要审批
- 违反项目规则
- 发布前缺少系统测试或 QA 结果

Gate 的目标是防错，不是接管执行。

### 4. 其他尽量留给 agent runtime + skill 提示词 + 项目产物

VibeFlow 应优先通过以下方式影响 agent：

- skill 中的方法论和执行顺序
- `rules/` 中的项目约束
- `design.md`、`tasks.md`、`feature-list.json` 等产物
- 当前项目的真实代码和验证输出

优先让 agent 在正确的上下文中工作，而不是把所有动作预先硬编码。

## 边界

VibeFlow 应负责：

- workflow phases
- contracts
- gates
- artifact layout
- recovery
- observability

VibeFlow 不应负责：

- 重新实现一个通用 agent runtime
- 重新实现一个通用 planner
- 重新实现一个通用 memory engine
- 重新实现宿主已经具备的多 agent 编排能力

## 设计判断标准

每当我们想新增一个脚本、状态机分支或自动化能力时，都先问四个问题：

1. 这件事是不是 100% 可机械化？
2. 这件事是不是必须稳定复现？
3. 这件事是不是经常忘，而且忘了会出事？
4. 这件事能不能更自然地由 agent runtime、skill 提示词或项目产物来承担？

如果前三个问题都答不强，优先不要下沉到框架层。

## 我们要建设什么

VibeFlow 要建设的是：

- 更清晰的阶段编排
- 更可靠的项目约束输入
- 更稳定的交付恢复能力
- 更真实的验证和发布闭环
- 更高质量的项目产物组织方式

## 我们不要建设什么

VibeFlow 不追求成为：

- 一个替代 Claude 或 Codex 的 agent
- 一个更重的自动化壳层
- 一个把所有判断都脚本化的系统
- 一个与宿主 runtime 竞争控制权的框架

## 一句话愿景

VibeFlow 不是替 agent 思考。

VibeFlow 是为 agent 提供正确的场地、边界、证据和节奏，让它用对方法完成交付。
