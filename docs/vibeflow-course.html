VibeFlow 是什么

一个让 AI 交付更有章法的编排框架

把一个想法，变成可交付的系统

VibeFlow 为 AI 辅助开发铺上轨道，让交付过程可追踪、可审查、可继续

大模型擅长即兴发挥，软件工程需要定力。真正稀缺的不是"生成一段代码"，而是从想法到发布的完整路径。 VibeFlow 做的就是给这条路铺上轨道。

💡
关键洞察

前 4 个阶段（Think → Design）需要你参与决策；后 4 个阶段（Build → Ship）可以由系统自动推进。这就是"前半程你拍板，后半程系统自己跑"的意思。

两种工作模式

🎯

Full Mode

完整的交付流程。适合新功能、架构改动、复杂需求。所有阶段都要走完，文档化程度高。

⚡

Quick Mode

快速通道。适合 bugfix、hotfix、小改动。限定范围，自动升级——如果发现范围超出预期，会提醒切回 Full。

📌
什么时候用哪个？

不确定时选 Full Mode。只有小范围、低风险、可快速回滚的改动（比如改个配置、修个 bug）才用 Quick Mode。

一个想法的旅程

1
你："给博客加个深色模式"

这句话进入 Think 阶段，VibeFlow 问你要覆盖哪些文件、支持什么场景、边界在哪。

2
Plan 阶段：价值评估

VibeFlow 帮你分析投入产出，问你："这值得花多少时间？用什么技术方案？"

3
Design 阶段：设计方案

生成具体方案文档——怎么改 CSS 变量、要不要改 JS 逻辑、影响范围多大。

4
Build 阶段开始 → 自动继续

设计方案确认后，系统自动推进 Build → Review → Test → Ship，你只需要在关键节点确认。

真实代码：状态读写

VibeFlow 用 JSON 文件而不是数据库来保存状态——这样任何人都可以直接查看和修改。

CODE 
def load_state(project_root):
    path = state_path(project_root)
    if not path.exists():
        return default_state(project_root)
    data = json.loads(path.read_text("utf-8"))
    merged = default_state(project_root)
    merged.update(...)
    return merged
PLAIN ENGLISH

Define a function to load the state file from disk.

If the file doesn't exist yet, return a default empty state.

Read the JSON file as text and parse it into a Python dictionary.

Start with default values, then overwrite with what's in the file.

Return the merged result — so the rest of VibeFlow always gets a complete state object.

💡
设计决策

VibeFlow 选择 JSON 文件而不是数据库，是因为文件可以 commit 到 Git、可以在任何编辑器打开、不会丢数据。即使 AI 崩溃，状态还在磁盘上。

03

文件即真相

状态如何持久化，checkpoint 机制是什么

state.json 里有什么

当你运行 /vibeflow 后，项目根目录会出现一个 .vibeflow/state.json 文件。这就是系统的"真相来源"。

.vibeflow/ VibeFlow 的系统目录
state.json 当前阶段、checkpoints、mode、change 信息
runtime.json 运行时状态、事件记录、友好提示
workflow.yaml 流程配置：哪些阶段必须手动、哪些可自动
packets/ 每个功能的执行数据包
subagent-results/ 功能执行结果（通过/失败）
docs/changes/ 这次变更的完整文档
context.md起点和目标
proposal.md方案提案
requirements.md正式需求
design.md实现方案
tasks.md执行清单

Checkpoint：进度快照

每个阶段完成后，会在 state.json 里打一个 checkpoint。就像游戏存档——崩溃了可以从最近的存档恢复。

CODE 
"checkpoints": {
  "think": true,
  "plan": true,
  "requirements": true,
  "design": true,
  "build_init": true,
  "build_config": false,
  "build_work": false,
  "review": false,
  "test_system": false,
  "ship": false
}
PLAIN ENGLISH

Here's the checkpoints section — a map of what we've finished.

Think, Plan, Requirements, Design — all done, marked as true.

Build-init is also done — the feature list was created.

Build-config is still false — we're just about to start it.

The rest (build-work, review, test, ship) are still pending.

📌
为什么这很重要？

如果你关掉 Claude Code 会话，下次回来时 VibeFlow 读取 state.json 就知道"我们已经完成了 Design，现在应该在 build-config"。它不会让你重新走一遍已经完成的阶段。

阶段检测逻辑

VibeFlow 有一个"阶段检测器"——每次 autopilot 想知道"下一步该做什么"时，就问它。它检查 checkpoint 和 artifacts 文件来决定当前应该处于哪个阶段。

CODE 
def detect_phase(project_root, ...):
    state = load_state(project_root)
    checkpoints = state.get("checkpoints", {})

    if not checkpoints.get("think"):
        return {"phase": "think", ...}
    if not checkpoints.get("plan"):
        return {"phase": "plan", ...}
    if not checkpoints.get("requirements"):
        return {"phase": "requirements", ...}
    if not checkpoints.get("design"):
        return {"phase": "design", ...}
    if not checkpoints.get("build_init"):
        return {"phase": "build-init", ...}
    ...
    return {"phase": "done"}
PLAIN ENGLISH

Define a function that looks at the checkpoints and tells you what the current phase is.

First check: is "think" marked done? If not, we're in the "think" phase.

Otherwise check "plan", then "requirements", then "design"...

Each one is a gate — you must pass through in order.

If all checkpoints are true, we're done — the whole workflow is complete.

并行构建：加速执行

当有多个功能（features）互相独立时，Autopilot 会并行执行它们——就像同时开多条生产线。

1
分析依赖关系

Autopilot 读取 feature-list.json，看哪些功能可以并行、哪些必须等前一个完成。

2
选择并行批次

找出文件范围不重叠的功能，组成一个批次同时跑。比如"加深色模式"和"改 SEO 标签"可以并行，因为它们改的是不同文件。

3
依赖满足后解锁

如果功能 B 依赖功能 A 的输出，那 B 必须等 A 完成后才能开始。

CODE 
def scopes_overlap(left, right):
    for l in left:
        for r in right:
            if l == r: return True
            if l.startswith(r + "/"): return True
            if r.startswith(l + "/"): return True
    return False
PLAIN ENGLISH

Function to check if two features touch overlapping files.

If both features modify the same file (exact match), they're not safe to run in parallel.

If one modifies a parent folder and the other modifies a subfolder, they also overlap.

Only if both conditions are false — completely separate paths — can they run at the same time.

数据如何流向 Dashboard

Dashboard 不是 VibeFlow 的一部分——它是一个独立的 HTTP 服务器，读取 state.json 和 runtime.json，然后在浏览器里渲染成好看的页面。

1
Dashboard HTTP 服务器

运行 python scripts/run-vibeflow-dashboard.py 启动，监听本地 4317 端口。

2
轮询 state.json + runtime.json

每 2 秒（或通过 Server-Sent Events 实时推送）读取最新状态。

3
构建快照

合并 state、runtime、feature-list、phase-history，计算每个阶段的状态（pending/running/completed/blocked）。

4
渲染 + 推送

生成 JSON 快照，通过 SSE（Server-Sent Events）推送给浏览器，浏览器更新 UI。

真实代码：Dashboard 快照构建

Dashboard 的核心是 build_dashboard_snapshot() 函数——它聚合所有数据源，返回一个完整的快照对象。

CODE 
def build_dashboard_snapshot(project_root):
    state = load_state(project_root)
    runtime = load_runtime(project_root)
    detect_info = detect_phase(project_root)
    flags = required_flags(contract["workflow"])
    feature_summary = load_feature_summary(project_root)

    for key, label, phases in PHASE_GROUPS:
        macro_cards.append({
            "key": key,
            "label": label,
            "status": build_macro_status(key, ...),
            "phases": [...]
        })

    return {
        "project": project_root.name,
        "current_phase": detect_info["phase"],
        "workflow": macro_cards,
        "features": feature_summary,
        "events": events,
        ...
    }
PLAIN ENGLISH

Define a function that collects all VibeFlow state into one snapshot object.

Read state, runtime, feature list, and detect the current phase.

Loop through all phase groups (think, plan, build, etc.) and calculate their status.

Return a dictionary with everything the dashboard needs to render — phase, features, events, and more.

常用命令速查

🚀

/vibeflow

开始或继续一个完整的工作流。第一次运行会选 Full/Quick 模式。

⚡

/vibeflow-quick

直接进入 Quick Mode，适合小改动。不确定时别用这个。

📊

/vibeflow-status

在聊天窗口里打印当前状态摘要，不需要开浏览器。

🖥️

/vibeflow-dashboard

打开本地浏览器看板，实时看进度、事件、feature 状态。

如何用它更好地指挥 AI

1
把需求说清楚，不要只说"加个功能"

VibeFlow 的 Requirements 阶段会问你：输入是什么、输出是什么、边界在哪。提前想清楚这些，你的 AI 会更准。

2
Design 阶段认真审阅

Autopilot 是根据 Design 执行 Build 的。如果设计里有模糊或错误的地方，Build 阶段就会出问题。

3
用 Dashboard 追踪进度，而不是盯着聊天窗口

Dashboard 告诉你当前在哪个阶段、哪个功能卡住了。如果你看到 blocked，去看 runtime.json 的 reason 字段找原因。

4
遇到问题先看 state.json

state.json 是系统真相。如果 autopilot 行为奇怪，先看看 checkpoint 是否被正确标记了。

💡
VibeFlow 的核心价值

它不只是让你少盯屏幕——更重要的是，它强迫你把需求想清楚、把方案设计明白、把验收标准定具体。这些"前期工作"做好了，后面的实现才不容易返工。

调试技巧

🔍

Autopilot 卡住了

检查 runtime.json 的 stop_reason 字段和 invariant.reason。Dashboard 的事件时间线也会显示最后发生了什么。

📄

Artifact 文件缺失

检查 .vibeflow/state.json 的 artifacts 字段，确认文件路径是否正确。可能是路径配置问题。

🔗

Feature 依赖没满足

检查 feature-list.json 里 dependencies 字段。被依赖的功能必须先标记为 passing，依赖它的功能才能开始。

⚙️

workflow.yaml 配置

控制哪些阶段是 manual_only（必须手动）、哪些是 auto_runnable（可自动）。如果某个阶段没有自动跑，检查这里。
